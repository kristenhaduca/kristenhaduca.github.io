Creating a School Database [SQL (SSMS)]

A database for a fictitious school is created by importing data from multiple Excel workbooks into SQL Server Management Studio. Data is cleaned and wrangled for clean formatting. Excessive, duplicated information between multiple tables are culled and only included in the most appropriate tables. Relationships between each entity are managed, additional lookup tables are created, and views and stored procedures are made for further reporting purposes.

PART 1: UNDERSTANDING THE DATA

The fictitious school has been in operation for about 8 years. Prior to now, they have been using Excel workbooks to track all information regarding the school and are now interested in using a database backend instead.
The data for the school was provided, consisting of two Excel workbooks that span about a 4 year duration each (SU11-SU15 and FA15-SU19). The school uses a quarter system for its enrollment.

Within each Excel workbook are 7 worksheets:

  • Sections - Contains information about each section of a course offered, including where and when a section is taught and instructor information for each section.
  • ClassList - Contains information about the students that attend each section, including contact information, grades, and amount of tuition paid.
  • Terms - Contains information about each term, including term codes and which year it takes place in.
  • Persons - Contains information about all students, including contact information and home addresses.
  • Rooms - Contains information about the school's rooms and capacity of each.
  • Courses - Contains information about the courses offered, including the time frame, tuition fee amount, and course description.
  • Faculty - Contains information about the school's faculty, including contact information and home addresses.

    • The primary relationships between the entities consist of:

  • A Section can belong to only one individual Course, but there can be multiple Sections within a single Course (e.g. the Course "Beginner I" can have multiple Sections: Section 10002, Section 10006, etc. Section 10002 can only correspond to the Course "Beginner I").
  • There can be multiple Faculty teaching an individual Section, and each Faculty member can teach multiple Sections (e.g. Section 10002 of "Beginner I" can be taught by multiple instructors, and instructor Octavia Livius can teach multiple Sections of "Beginner I").
  • There can be multiple Persons (students) within the same Section, and each Person (student) can be enrolled in multiple Sections (e.g. multiple students can be enrolled in Section 10002, and student Abel Richardson can enroll in both Section 10198 and Section 10233).

    • The school currently only stores a single address for each person, but they would like to include the option to add both a Home and Work address in the future. Any existing address is considered a Home address.

    FacultyPayment Table

    Two new tables, FacultyPayment and Address, are also created to house important information.
    FacultyPayment lists the instructors that teach each section along with their payment amounts, and is required to join the Faculty table with the Sections table.

    First, the new table, FacultyPayment, is populated using data from the Faculty and Sections tables with SELECT ... INTO.
    The Sections table has the faculty names listed as the first initial of the first name, followed by a period, then the last name.
    The Faculty table has the faculty names listed as the first name and the last name as two separate columns.
    CONCAT() is used to make Faculty's name scheme the same as Sections in order to JOIN on the PrimaryInstructor column. LEFT() is used to extract the first character starting from the left-hand side of FacultyFirstName.

    SELECT
          Sections.SectionID
          ,Sections.PrimaryInstructor
          ,Sections.PrimaryPayment
          ,Faculty.FacultyID
          ,Sections.SecondaryInstructor
          ,Sections.SecondaryPayment
    INTO FacultyPayment
    FROM Sections
    LEFT JOIN Faculty
          ON CONCAT(
                LEFT(FacultyFirstName, 1),
                '. ',
                FacultyLastName
          )
          = Sections.PrimaryInstructor
    ORDER BY SectionID

    Because there is a PrimaryInstructor and SecondaryInstructor differentiation, but the Faculty table only has a single FacultyID column, the previous query's FacultyID column only represents the PrimaryInstructor's ID. To list SecondaryInstructor's ID, a new column called SecondaryFacultyID is made, then it is populated by the FacultyID corresponding to the name under SecondaryInstructor.

    ALTER TABLE FacultyPayment
          ADD SecondaryPaymentID INT

    UPDATE FacultyPayment
          SET SecondaryPaymentID = Faculty.FacultyID
    FROM FacultyPayment
    JOIN Faculty
          ON CONCAT(
                LEFT(FacultyFirstName, 1),
                '. ',
                FacultyLastName
          )
          = FacultyPayment.SecondaryInstructor

    PART 5: CREATING VIEWS AND STORED PROCEDURES

    1. VIEW - Number of Times Course Has Been Offered And Its Revenue

    A view is created that displays how many times a course has been offered over the entire history of the school, the total gross revenue of each course, and the average revenue garnered per section of a course. Cancelled sections (SectionStatus = CN) are not included.

    SECTION COUNT

    To find the section count of a course, the SectionCount, COUNT() is used in tandem with DISTINCT in order to count each distinct SectionID that has been offered.

    COUNT(DISTINCT Sections.SectionID) AS SectionCount

    TOTAL GROSS REVENUE

    To find the total gross revenue of each course, the TotalGrossRevenue, the SUM() of the TuitionAmount column within the ClassList table is calculated. Later in the query, everything is grouped by the CourseCode and the CourseTitle, so the sum is grouped within each individual course.

    SUM(ClassList.TuitionAmount) AS TotalGrossRevenue

    To see this illustrated further, we look at Sections.CourseCode = LAT-108. Each Course can have multiple Sections; within LAT-108, there are two SectionIDs: 10968 and 11216. Within each Section, there are multiple Persons, each with their own tuitions paid. SUM() adds up all TuitionAmounts of every Person within each Section of a single Course.

    AVERAGE REVENUE PER SECTION

    The AverageRevenue is found by dividing the SUM() of TuitionAmount within each class (a.k.a. the TotalGrossRevenue) by the COUNT() of distinct SectionIDs. CAST(... AS NUMERIC(9,2)) is used to allow this value to be listed as a numeric with 9 total digits and 2 decimal places.

    CAST(
          SUM(ClassList.TuitionAmount)
          /
          COUNT(DISTINCT Sections.SectionID)
    AS NUMERIC(9,2)
    ) AS AvgRevenue

    FINAL CODE

    GO
    CREATE VIEW CourseRevenue_v AS
    SELECT
          Courses.CourseCode
          ,Courses.CourseTitle
          ,COUNT(DISTINCT Sections.SectionID) AS SectionCount
          ,SUM(ClassList.TuitionAmount) AS TotalGrossRevenue
          ,CAST(
                SUM(ClassList.TuitionAmount)
                /
                COUNT(DISTINCT Sections.SectionID)
          AS NUMERIC(9,2)
          ) AS AvgRevenue
    FROM Courses
    LEFT JOIN Sections
          ON Sections.CourseID = Courses.CourseID
    LEFT JOIN ClassList
          ON ClassList.SectionID = Sections.SectionID
    WHERE Sections.SectionStatus != 'CN'
    GROUP BY Courses.CourseCode, Courses.CourseTitle

    2. VIEW - Gross Tuition Revenue and Faculty Salary Payments

    A view is created in order to calculate the gross tuition revenue and faculty salary payments for each academic year.

    CREATING CTE

    A CTE is created that selects SectionID and the TotalTuition. TotalTuition is the sum of all TuitionAmounts, grouped by their SectionIDs (a.k.a. the total tuition paid per individual section).

    SELECT
          Sections.SectionID
          SUM(ClassList.TuitionAmount) AS TotalTuition
    FROM Sections
    JOIN ClassList
          ON ClassList.SectionID = Sections.SectionID
    GROUP BY Sections.SectionID

    CREATE TOTALPAYMENT COLUMN

    In the FacultyPayment table, payments to the faculty are broken up into two columns, PrimaryPayment and SecondaryPayment. In order to get the full payment total for each faculty member, a new column called TotalPayment is created. SET() is used to update the new TotalPayment column to be the added amounts of PrimaryPayment and SecondaryPayment. COALESCE() is wrapped around SecondPayment to say if SecondaryPayment is a NULL value, it should be used as a 0 value instead.

    ALTER TABLE FacultyPayment
          ADD TotalPayment INT

    UPDATE FacultyPayment
    SET TotalPayment =
          (PrimaryPayment +
                COALESCE(SecondaryPayment, 0)
          )

    FINAL CODE

    The Q2_CTE.TotalTuition column from the CTE (total tuition paid per section), grouped by AcademicYear, is summed—resulting in the total tuition paid per academic year. The FacultyPayment.TotalPayment column, grouped by AcademicYear, is summed—resulting in the total faculty payments per academic year.

    GO
          CREATE VIEW AnnualRevenue_v AS
    WITH Q2_CTE AS
    (
    SELECT
          Sections.SectionID
          ,SUM(ClassList.TuitionAmount) AS TotalTuition
    FROM Sections
    JOIN ClassList
          ON ClassList.SectionID = Sections.SectionID
    GROUP BY Sections.SectionID
    )
    SELECT
          Terms.AcademicYear
          ,SUM(Q2_CTE.TotalTuition) AS TotalTuition
          ,SUM(FacultyPayment.TotalPayment) AS TotalFacultyPayment
    FROM Courses
    JOIN Sections
          ON Sections.CourseID = Courses.CourseID
    JOIN Terms
          ON Terms.TermID = Sections.TermID
    LEFT JOIN FacultyPayment
          ON FacultyPayment.SectionID = Sections.SectionID
    LEFT JOIN Q2_CTE
          ON Q2_CTE.SectionID = Sections.SectionID
    GROUP BY Terms.AcademicYear

    3. STORED PROCEDURE - Course History of a Student

    A stored procedure is created that shows the course history of a student when supplied with the primary key number (a.k.a the StudentID) of the student.
    CASE() is used to say if there is a middle name supplied for the student, StudentName will be written as "FirstName MiddleName LastName", else it will be written as "FirstName LastName".

    GO

    CREATE PROCEDURE StudentHistory_p
          @StudentPK INT
    AS
    SELECT
          CASE
                WHEN Persons.MiddleName IS NOT NULL THEN CONCAT(Persons.FirstName, ' ', Persons.MiddleName, ' ', Persons.LastName)
                ELSE CONCAT(Persons.FirstName, ' ', Persons.LastName)
                END AS StudentName
          ,ClassList.SectionID
          ,Sections.CourseCode
          ,Courses.CourseTitle
          ,Sections.PrimaryInstructor
          ,Terms.TermCode
          ,Sections.StartDate
          ,ClassList.TuitionAmount
          ,ClassList.Grade
    FROM Persons
    JOIN ClassList
          ON ClassList.PersonID = Persons.PersonID
    JOIN Sections
          ON Sections.SectionID = ClassList.SectionID
    JOIN Courses
          ON Courses.CourseID = Sections.CourseID
    JOIN Terms
          ON Terms.TermID = Sections.TermID
    WHERE Persons.PersonID = @StudentPK
    ORDER BY Sections.StartDate

    GO
